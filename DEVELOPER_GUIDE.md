# Getting Started with Go Module Development

This guide explains how to work with the files generated by `viam module generate` for Go modules.

---
## Fast local development (`salad-cli app`)

A module reload takes ~2 minutes and forces you to debug via live machine logs.
`salad-cli app` runs the salad module **locally** against a live machine instead,
so the loop is: edit a `.go` file → `go build ./cmd/cli` (seconds) → run, with
breakpoints and logs on your laptop.

```bash
make app                       # build salad-cli + run; REPL against salad1
make app ARGS="--simulate"     # run builds without moving hardware
make app ARGS="--exec status"  # run one command and exit (good for scripts)
make app ARGS="--debug"        # verbose (rdk client + service debug logs)
```

### How it works

`salad-cli app` resolves the machine from a robot id, dials it for hardware,
pulls the fully fragment-merged config via your `viam login` token, then
constructs **every salad module resource** (coordinator + grabber, dressing,
chefs-kiss, supply-detector, file-vision, maintenance-sensor) **in this process**
using the same constructors `cmd/module` registers. Resources are built in
dependency order and injected into the dependency map so each binds to the local
copy. Everything that isn't a salad model — arms, grippers, cameras, switches,
motion, and other modules' services — stays on the machine and is reached as a
remote resource.

If a resource can't build locally (e.g. `file-vision` needs a mesh file that only
exists on the machine), it logs a warning and falls back to the remote one; the
coordinator failing is fatal.

### REPL

```
build lettuce=2 chicken=1 [customer=Alice]   # drive a build through the coordinator
status | stop | setup | ingredients | theme  # coordinator shortcuts
do <resource> {json}                         # send a raw DoCommand to any local resource
resources                                    # list local resources + their DoCommand verbs
fetch-assets                                 # re-pull the machine's real setup assets
events                                        # drain queued build-events
help | quit
```

`do` is how you exercise a single sub-service, e.g.
`do dressing-controls {"get_dressings": true}`. Run `resources` to see what's
local and which verbs each model accepts.

### Flags

- `--machine <robot-id>` — defaults to salad1; address, part id and secret are
  derived from it. Requires `viam login`.
- `--simulate` — sets the coordinator's simulate flag (build flow + events, no
  motion). **Default is real hardware** — `build` moves the arm.
- `--work-dir <dir>` — local dir (default `./.salad-local`) for setup/build
  assets, so the coordinator never touches `/home/viam` or `/root`. Backed by the
  configurable `assets-dir`/`capture-dir`; on the machine these still default to
  their original paths.
- `--wire-events` (default on) — constructs a local in-memory `build-events`
  sensor and wires it into the coordinator if the machine has none. **Events are
  debug-only**: drained in-process and printed; no data manager runs locally, so
  nothing is captured or synced.
- `--debug` — verbose logging.

### Notes & safety

- The machine's own deployed coordinator is still live. Don't trigger a build
  from the web app while driving here — two coordinators would fight over the arm.
- Ctrl-C stops any in-flight build and resets hardware before exiting.
- A `build` uses the machine's **real** setup data: the first build (or
  `fetch-assets`) copies `merged.pcd`, `zones.json`, and `mesh.ply` from the
  machine into `--work-dir` via the machine's shell service (`viam machines part
  cp`), and points the coordinator + grabber + dressing at that dir. Nothing is
  generated locally and no module deploy is needed (requires `viam` on PATH and a
  shell service on the machine; override the source path with
  `--machine-assets-dir`). Running `setup_station` locally (the Python meshifier)
  is the deferred follow-up; until then, `setup` is run on the machine.

### Breakpoints (VS Code / Delve)

Drop this into `.vscode/launch.json` (gitignored) to run under the debugger:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "salad app (local)",
      "type": "go",
      "request": "launch",
      "mode": "debug",
      "program": "${workspaceFolder}/cmd/cli",
      "args": ["app"],
      "console": "integratedTerminal"
    }
  ]
}
```

Or headless: `dlv debug ./cmd/cli -- app`.

## Working with `module.go` and module lifecycle

The generated `module.go` file contains the scaffolding for your resource. You will edit this file to add configuration, initialization, and method implementations.

1. **`Validate(path string)`**
   - Called first, before the module is created.
   - It checks that the **Config struct** (`cfg`) for this specific resource is valid.  
   - You can use it to ensure required fields exist and values are within valid ranges.  
   - It also returns dependency information:  
   - **First return:** required dependencies (other resources that must exist)  
   - **Second return:** optional dependencies (resources that are nice to have but not required)  
   - `path` is provided by the runtime and indicates **where this resource appears in the JSON config**, e.g., `"components.0"`. Use it in error messages to show which resource has a problem.

Example:

```go
func (cfg *Config) Validate(path string) ([]string, []string, error) {
    if cfg.Camera == "" {
        return nil, nil, fmt.Errorf("%s: missing required field 'camera'", path)
    }
    return []string{"board:main"}, []string{"sensor:temp"}, nil
}
```

2. **`new<ModuleName><ModelName>(...)`**
   - Automatically called by the runtime after validation.
   - Takes a generic `rawConf` from the robot’s JSON config.  
   - Converts it into a typed `*Config`.
   - Passes the config to `New<ModelName>` to create the resource instance.
   - You usually won’t edit this function. 

3. **`New<ModelName>(...)`**
   - Your actual constructor.
   - Receives the already-parsed `*Config`. 
   - Sets up the logger, config, and any initial state.
   - You can add hardware initialization or background tasks here.
   - Returns the fully constructed resource instance.

4. **Resource Methods**
   - These are called based on what is being used in the robot.

   - The generated file contains stubs for all required interface methods. Each currently calls `panic("not implemented")`.  

   - Replace these with your logic. For example:

```go
func (r *myResource) DoCommand(ctx context.Context, cmd map[string]interface{}) (map[string]interface{}, error) {
    r.logger.Infof("DoCommand called with: %+v", cmd)
    return map[string]interface{}{"status": "ok"}, nil
}
```

5. **`Close(ctx)`**
   - Called when the module is being shut down.
   - Use it to clean up background tasks, stop goroutines, and close hardware connections.


### Logging

Use the provided `logger` for all debug, info, and error messages. You can view these logs on the Viam app when you're module is running on any robot.

Examples:

```go
// Informational message
s.logger.Infof("Starting move sequence for arm %s", s.name)

// Debug message
s.logger.Debugf("Current joint positions: %+v", positions)

// Error message
s.logger.Errorf("Failed to move resource: %v", err)

// Example in a method
func (s *<ModuleName><ModelName>) EndPosition(ctx context.Context, extra map[string]interface{}) (spatialmath.Pose, error) {
    s.logger.Infof("EndPosition called for resource %s", s.name)
    return spatialmath.Pose{}, nil
}
```
